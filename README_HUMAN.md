## Как была организована работа

Весь код написан через AI-агентов — задание совпало с моментом, когда я осваивал новый тулсет после перерыва от индустрии.

Использовал два инструмента:

- **[CC Thingz](https://github.com/umputun/cc-thingz)** — плагины для Claude Code, в частности brainstorm и planning. Использовал для проработки задач перед передачей в Ральфикс.
- **[Ralfix](https://github.com/umputun/ralphex)** — система агентов поверх Claude Code (разработчики + ревью). Получал план задачи и отдавал его Ральфиксу, который автономно генерировал код.

Из CC Thingz использованы скиллы brainstorm и planning.  
Алгоритм: brainstorm → зафиксированный план → автономная генерация через Ральфикс. Планы хранятся в `docs/plans/` (выполненные в `docs/plans/completed/`). По соглашениям проекта Ральфикс планы хранятся там, поэтому отдельную папку `.agents` не заводил.

Моя роль: проектирование, постановка задачи, ревью результата и принятие архитектурных решений. Код я не писал руками, но большинство решений в нём — осознанные.

## Язык и стек

Выбрал Python, потому что у меня есть продакшн-опыт на нём, и я могу критически оценить каждое решение в коде — не только структурно, но и по сути. Если бы писал на Go или TypeScript, код выглядел бы правдоподобно, но я не мог бы честно сказать, что уверен в каждом нюансе.

FastAPI — очевидный выбор для Python-сервиса с async: зрелый, с хорошим DI, lifespan-контекстом и нормальной экосистемой библиотек. Ниже уровнем можно было взять aiohttp, но там потерял бы инструментарий без реального выигрыша.

PostgreSQL — отраслевой стандарт для реляционных данных, здесь нет никакого выбора по сути.

Redis — когда нужен быстрый in-memory store, Redis — самый зрелый вариант в индустрии.

## GraphQL, а не gRPC и не REST

REST не даёт структурного разделения команд и запросов — его приходится имитировать конвенциями. Для задания с явным CQRS это добавляет лишний шум.

gRPC хорош для межсервисного взаимодействия, но браузер не умеет говорить gRPC без прокси. Для auth-сервиса, который часто является первой точкой входа для веб-клиентов, это лишняя инфраструктура.

GraphQL — хороший компромисс: mutations и queries структурно закреплены в схеме, браузер работает напрямую, Strawberry позволяет использовать Python-датаклассы как GraphQL-типы. Доменные объекты описываются один раз и без отдельного .graphql файла, который нужно держать в синхронизации.

Дополнительный бонус — встроенный Playground на `/graphql`: ревьюер может сразу потрогать API без Postman.

## Архитектура: слоёный DDD, не гексагональный

DDD здесь — не ради формальности. Три потока в задании (регистрация, логин, восстановление пароля) имеют конкретные бизнес-правила: когда токен сброса считается использованным, в каком порядке проверяются инварианты, что значит «неактивный аккаунт». Всё это должно жить в доменном слое, а не в ORM-моделях или резолверах.

Гексагональная архитектура добавляет церемонию: явные первичные порты (интерфейсы use-case), которые реализуют хендлеры. При шести хендлерах и трёх флоу это чистый оверинжиниринг — каждый use-case существует в одном экземпляре, тестовых дублей для первичных портов нет. Вторичные порты (репозитории, хешер, токен-сервис, email) — вот где инверсия зависимостей реально нужна, и они есть в `application/ports/`.

Итог: domain ничего не знает про SQLAlchemy и Redis, application ничего не знает про инфраструктуру, всё вместе собирается в `container.py` — явно, без магии фреймворка.

## Механизм сессий и JTI

Для финтех-продукта критически важно уметь мгновенно отозвать токен: скомпрометированный аккаунт, сигнал от антифрода, принудительный разлогин. Чисто stateless JWT это не позволяет — нужно ждать истечения.

Решение: короткоживущие access-токены (15 минут) с `jti` (JWT ID), хранящимся в Redis как allowlist. При проверке токена делается один Redis-запрос — если `jti` нет в хранилище, токен отозван. Allowlist ограничен числом активных сессий; blocklist рос бы неограниченно.

Refresh-токены — непрозрачные UUID в Redis, привязанные к сессии. При рефреше — `GETDEL`: атомарно читаем и удаляем, выдаём новый токен. Один refresh-токен — один раз.

Нюанс, который я пропустил при составлении плана: все данные о сессиях живут только в Redis. Если нужно показывать пользователю список активных устройств или хранить device ID — метаданные сессий лучше держать в PostgreSQL. Обнаружил это после генерации, но переделывать не стал — лишние токены, функциональности это не меняет. В продакшн-версии это нужно пересмотреть.

## Идемпотентность

Покрыл два метода: `login` и `requestPasswordReset`.

Остальные не нуждаются: `register` вернёт ошибку на дублирующий запрос, `resetPassword` — токен одноразовый и `consume()` бросит ошибку повторно, `refreshToken` — `GETDEL` атомарный, `revokeSession` — повторный отзыв несуществующей сессии уже no-op.

Механизм: заголовок `Idempotency-Key` + SHA-256 от операции и тела запроса. Кэш в Redis на 24 часа. Если пришёл тот же ключ с другим телом — GraphQL error `IDEMPOTENCY_CONFLICT` (HTTP 200, потому что в GraphQL ошибки всегда в `errors[]`).

Реализовано через Strawberry `SchemaExtension`, чтобы не трогать хендлеры.

## Rate limiting

Реализован как ASGI-middleware, который перехватывает запросы до GraphQL-резолвера. Читает тело запроса, извлекает имя операции и email из переменных, затем применяет лимиты через Redis.

Механизм — fixed-window counter на Redis: `SET NX EX` инициализирует ключ с TTL атомарно, затем `INCR` увеличивает счётчик. При превышении — HTTP 429 с заголовком `Retry-After`. Sliding window был бы точнее, но для защиты от брутфорса fixed window достаточен и значительно проще.

Лимиты двумерные — по IP и по email одновременно — потому что это разные векторы атак:

| Операция | Измерение | Лимит | Окно |
|---|---|---|---|
| `login` | IP | 5 | 1 минута |
| `login` | email | 10 | 15 минут |
| `register` | IP | 5 | 1 час |
| `requestPasswordReset` | email | 3 | 1 час |

IP определяется из заголовка `X-Real-IP` (выставляется доверенным прокси), но не из `X-Forwarded-For` — его клиент может подделать и обойти IP-лимит, выставив произвольный адрес.

**Где должен быть rate limiting в продакшне.** Правильно — снаружи приложения: firewall, балансировщик, API-gateway. Они видят весь трафик и могут отрезать по IP ещё до приложения. Но может понадобиться более умный лимитинг — завязанный на device-метаданные или паттерны поведения конкретного пользователя, а не только на IP. Такое снаружи не сделать. В рамках продакшн-разработки мы бы обсудили конкретные abuse-кейсы и определили границу между внешним и внутренним слоем защиты.

**Почему нет жёсткой блокировки аккаунта.** Hard lockout создаёт DoS-вектор: атакующий, зная email жертвы, намеренно блокирует её аккаунт. Rate limiting по IP/email даёт backoff без этого риска.

## Аудит

Таблица `auth_events` в PostgreSQL — append-only лог всех событий: логин, неудачная попытка, разлогин, сброс пароля. Redis-данные эфемерны и TTL-управляемы; аудитная таблица — durable record для compliance.

Важное архитектурное правило: вызов `AuditLogPort.record()` всегда обёрнут в `try/except Exception`. Сбой логирования не должен ронять auth-операцию.

## Password reset

`requestPasswordReset` ищет пользователя по email и, если не находит, явно возвращает `success=False`. Защита от email enumeration здесь намеренно убрана — продуктовое решение в пользу понятного UX. (Согласно макету. Интуитивное решение - выдавать success на несуществующий email). Если пользователь найден — инвалидируем предыдущие reset-токены, генерируем `secrets.token_urlsafe(32)` и отправляем его на email. В БД сохраняется только SHA-256 хэш — raw токен не персистируется, утечка БД не компрометирует токены. TTL — 15 минут.

Текущая реализация удаляет предыдущие токены из БД (`delete_all_by_user_id`) — это самодеятельность планировщика, не прописал явно при обсуждении. Правильнее помечать их как деактивированные, сохраняя историю запросов на сброс пароля.

`resetPassword` сначала валидирует новый пароль до обращения к БД, затем хэширует входящий токен SHA-256 и ищет по хэшу. Доменный метод `consume()` проверяет два инварианта в жёстком порядке: сначала TTL, потом одноразовость — порядок осознанный, просроченный токен должен сообщать именно об истечении срока, а не прятаться за «уже использован». После смены пароля — ревокация всех активных сессий пользователя, форс-релогин.
